## 概念

### 什么是集群（Cluster ）

集群（也称：群集）：将**多台计算机组合**在一起，来**完成特定的任务**。

为什么需要集群？

**1. 高性能计算**

利用多台相对计算能力低的计算机的并行计算能力，从而获得相当于超级计算机的计算能力。

一般是将大量的计算任务和数据分成小块，这些块和数据由多台计算机分别计算. 上传计算结果后，将结果统一并合并以获得结果。如：基因序列测算、能源勘测、芯片设计等行业

**2. 高可用性、高可伸缩性**

以前的单机架构（所有的应用服务、数据服务等都放在一台机器上）面对大流量时不再使用，往往会因为整个服务链中某一环断掉而导致所有服务受损。

集群方式部署可以有效解决 “鸡蛋放在同一个篮子里” 的问题。

谈到集群或多或少会提到：分布式、主从、主备这些字眼，这里记录一下自己的理解。

### 什么是有状态、无状态

从上述集群、分布式的描述可以看出来它们都是为了保证服务的“高可用”这一目的。对于“有状态服务”来说，那到底怎么做呢？

主从、主备这两种较为较为常见的**集群做事方式**。

说主从、主备之前需要说一下：什么是“无状态服务”、“有状态服务”？

#### 无状态服务

在不同时刻，服务不会有多个版本的差异（**即：不管什么时候提供的服务都是无差异的**）。

譬如：以 Nginx 为例不管谁来访问、不管什么时候来访问，那么 Nginx 本身所能提供的服务都是一样，它不应该也不会因为用户的行为差异、时间差异而导致服务本身出现版本差异的。

#### 有状态服务

在不同一时刻，服务本身可能会有不同的状态差异。

譬如：文件服务或数据库服务，由于用户上传了一份文件、删除了一份问题、用户的注册行为、用户的注销行为会导致在不同时刻查询文件列表、数据库列表所得到的服务结果会有变化。

### 集群的几种模式

“集群”简单的说就是：用一堆机器完成同一类事情。

而这一堆机器内部可以有多种方案来达到这一目的：集群模式（流量大）、主从模式（读写分离问题）、主备模式（单节点故障问题）。

#### 集群模式

该模式的架构如下图解：

![一般的高可用集群架构](https://minsonlee.github.io/images/pig/20211016114418.png)

**注意：**

1. 这里的一个节点不一定就是代表一台服务器
2. 各节点之间不一定都是同一个状态
3. 之前我一直不理解以上的模式中，由于所有流量都从“负载均衡”的机器进来，那“负载均衡”的机器不是就容易挂吗？

答案是：否定的。因为在实际生产中“负载均衡”的那台机器是不会参与任何复杂计算的单纯只是做流量转发，所以一般来说都是不需要太高配且不容易出错的。

而将流量从“单入口”的模式引进也方便了对流量进行管理，譬如：发现有异常的 IP，只需要在负载均衡的机器进行配置拦截即可，不需要在后面每台机器上都重复一次操作；后续增加或删除节点时，对于上层应用来说都是无感知的。

------

#### 主从集群模式

主从模式（Master-Slave）又称为：主仆模式。

**一般来说主节点是负责分配工作+整合结果，从节点负责执行具体的工作任务，且从节点只能和主节点之间通讯**。

但是在实际生产中，主从模式更多被用于解决：读写分离的问题（查询压力过大）。即：主节点可能会有压力小的工作。

基本所有的操作都可以归纳为：读、写两种操作。

- 读：查询数据
- 写：新增数据、更新数据、删除数据

对于大多数的应用来说：不管从数据操作的频率、数据操作的复杂性来说，读数据（查询操作）的肯定是比写数据要高频、复杂的。即：**读的压力比写的压力大**。

利用主从模式解决读写分离（一主一从模式）如下图解：

![what is Master-Slave](https://minsonlee.github.io/images/pig/master-slave.png)

**注意：**
1. **无三不成多**：一般来说主从模式都会有3个节点（一主两从）来保证节点的高可用性，因为一般来说要 3 个节点同时出问题，这个概率是非常小的。
2. **不一定是主写从读**：同步数据是需要时间的，因此主、从节点之间必然会存在数据延迟问题（即：同一时刻从节点数据落后于主节点）。所以，做读写分离时，业务程序上一定要分辨清楚当前操作对数据的时效性。对于有实时要求的操作那么读数据时也需要走主节点的。

![一主两从](https://minsonlee.github.io/images/pig/one-master-two-slave.png)

-------------

#### 主备集群模式

主备模式如下图解：

![主备模式](https://minsonlee.github.io/images/pig/20211017031348.png)

主备模式用于解决：单点故障问题。即：主节点同时提供读写操作，备份节点只负责同步数据且备份节点只有在主节点挂了之后才会被使用（替换故障的主节点成为新的主节点，继续提供）。


---------



对于**无状态服务**来说，由于提供的服务不管在什么时刻都是无差异化的，因此：为了**缓解流量过大**的问题，一般无状态服务使用集群都只需要**横向扩展机器**来分担流量压力就可以了。因此，**对于无状态服务来说，集群大多数情况下都是：一个节点就一台机器、每个节点的状态都一样**。

对于**有状态服务**来说（一般都是数据库服务），这些情况就复杂的多了。如：MySQL 主从复制、Redis 的集群/主从/哨兵模式、ES 的集群等问题。

------

### 什么是分布式？

分布式服务：将一个大的系统服务拆分为多个子系统服务，从而降低系统之间的耦合来保证不会因为服务中某一环节不可用时导致整个服务不可用（高可用性），同时也提高了服务的可复用性。如下图解：当优惠券服务宕机了，至少用户还能正常搜索、注册登录等行为；同时优惠券系统也可以作为单独服务提供给其他的系统使用。

![分布式服务](https://minsonlee.github.io/images/pig/20211016120804.png)

**分布式系统的目的：同一系统间高内聚，不同系统之间低耦合**。

#### CAP 理论

CAP 定理（CAP theorem），又被称作布鲁尔定理（Brewer's theorem）：


选项 | 描述 | 是否必须
---|---|---
Consistency（一致性） | 指数据在多个副本/节点之间能够保持严格的一致性 | 否
Availability（可用性） | 指服务一直出于可用的状态，每次请求都能获取到非错的响应（不保证获取的数据为最新数据）| 否
Partition tolerance（分区容错性）| 分区相当于对通信的时限要求。系统如果不能在时限内达成数据一致性，就意味着发生了分区的情况，必须就当前操作在C和A之间做出选择 | 是

根据定理，分布式系统只能满足三项中的两项而不可能满足全部三项。但实际中，既然是分布式系统就一定要保证 “P-分区容错性” 是成立的，因此也就剩下：C、A 的选择问题。

对于“C-一致性问题”，有3个标准：强一致性、弱一致性、最终一致性。


选项 | 标准
---|---
强一致性 | 写入的数据能迅速被读取到
弱一致性 | 写入的数据能容忍部分或全部不被读取到
最终一致性 | 写入的数据**经过一段时间后**要求能被**全部访问**到


对于 “A-可用性”，那么上述说的诸多“集群模式”就是高可用的解决方案。


一般来说，分布式都会依赖集群的（因为服务越多、维护的成本其实也就越大），如果服务已经到了需要走分布式了那么大概率一定是已经经历过了单服务集群的时代了。

在设计分布式系统时，应考虑以下几个问题：
- 系统如何拆分为子系统？
- 如何规划子系统间的通信？
- 通信过程中的安全如何考虑？
- 如何让子系统可以扩展？
- 子系统的可靠性如何保证？
- 数据的一致性是如何实现的？

### 概念总结

**“分布式”相当于：一个公司分成多个部门**，每个部门**各司其职（各部门负责不同的事但可能又有相互往来）** ，从而达到提高**公司（服务）的稳定和高效**的目的；“集群”相当于：一个部门聘请多个员工同时**干一类事情**，从而**提高稳定和效率**。

不管是分布式、集群、还是 主从、主备 ，这些都不是单一存在的。在一个复杂的系统中，往往是需要根据不同的业务场景选择不同的方案交替存在的。

(4). 配置网络

`CentOS 7` 通过 `ip addr` 查看网卡信息，可以通过该命令看一下默认网卡名字，然后在 `/etc/sysconfig/network-scripts` 目录下找到对应的网卡配置文件。

![ip addr](https://minsonlee.github.io/images/pig/20211017110024.png)

先查看一下本机的网关等信息，有两种方式可以查看。

(a). 在宿主机（Windows）中通过 `ipconfig` 查看当前无线网卡的信息。

![cmd ipconfig](https://minsonlee.github.io/images/pig/20211017110911.png)

(b). 通过 DHCP 先自动给虚拟机分配一个 IP 地址信息

```
vi /etc/sysconfig/network-scripts/ifcfg-enp0s3

# 设置网卡启动时通过动态分配一个 ip 地址（将文件中的两项配置做一下修改）
BOOTPROTO=dhcp
ONBOOT=yes

# 重启网卡：CentOS8 重启网卡是 `nmcli c reload`
service network restart

# 此时可以查看当前自动分配的网址信息
# 根据 `192.168.31.100/24` 即最后一段地址是可变更的
# 那么网关（192.168.31.1）和子网掩码（255.255.255.0）就可以知晓
ip addr
```

重新编辑对应的网卡配置文件，为其设置静态 IP 地址。修改 `BOOTPROTO` 类型为静态IP-`static`

```text
TYPE=Ethernet // 网络类型
BOOTPROTO=static // 网卡获取ip地址方式：none dhcp static
DEFROUTE=yes // 默认路由
IPV4_FAILURE_FATAL=no // 
IPV6INIT=yes
IPV6_AUTOCONF=yes
IPV6_DEFROUTE=yes
IPV6_FAILURE_FATAL=no
IPV6_ADDR_GEN_MODE=stable-privacy
NAME=enp0s3 // 网卡名称（没懂跟 DEVICE 有啥区别）
UUID=522b7e89-98a8-4d9e-8b34-3dc0be2c43d5 // 网卡唯一标识
DEVICE=enp0s3 // 网卡名称
ONBOOT=yes // 开机或者重启是否重启网卡
IPADDR=192.168.31.100 // 静态 IP 地址
PREFIX=24 // 网段
GATEWAY=192.168.31.1 // 设置网关
DNS1=114.114.114.114 // 设置电信的 DNS 解析服务地址
DNS2=8.8.8.8 // 设置谷歌 DNS 解析服务地址
IPV6_PEERDNS=yes // 解析 IPV6 域名
IPV6_PEERROUTES=yes // 开启 IPV6 地址
```

```sh
# 重启网卡：CentOS8 重启网卡是 `nmcli c reload`
service network restart

# 查看当前自动分配的网址
ip addr
```

配置DNS

- 检查 NetManager 的状态：systemctl status NetworkManager.service
- 检查 NetManager 管理的网络接口：nmcli dev status 
- 检查 NetManager 管理的网络连接：nmcli connection show
- 设置网卡的 DNS 信息：nmcli con mod enp0s3 ipv4.dns "114.114.114.114 8.8.8.8"
- 让 DNS 配置生效：nmcli con up enp0s3

### 3、配置节点之间 ssh 免密通信

```
# 生成密钥
 ssh-keygen -t rsa -C "master/node1/node2/node3"

# 将公钥上传到服务器，此时需要输入密码的
ssh-copy-id -i ~/.ssh/id_rsa.pub root@remote-ip-or-address

# 上述的命令等同于三个步骤
# 1. 将公钥上传至
scp id_rsa.pub  <user>@<remote-ip-or-address>:/tmp
# 2. 将公钥添加到 许可密钥（authorized_keys） 文件中
cat /tmp/id_rsa.pub >> ~/.ssh/authorized_keys
# 3. 删除公钥文件
rm  /tmp/id_rsa.pub
```
