## 使用上下文对象传参

### 问题1：“上下文”对象跟形参列表方式传参的区别

问题 1、这个context跟我直接 obj里面写obj1 obj2有嘛区别？感觉就是对象套对象，变成一个大而全的对象，没啥区别...

首先，“上下文”和“对象套对象，变成一个大而全的对象” 两者的概念是不一致的。

- “上下文” 强调的遇到这种复杂业务场景，数据间的传递应当用“上下文”的这种思维概念来处理
- “对象嵌套对象” 是一种具体的做法，是构建出上下文对象常用的一种承载实现手段

### 问题2、有什么优点？

当业务中数据对象很多时候，如果全部显示传参`(obj1, obj2, obj3...)` 确实可以感受到所谓的 “细粒度” 所带来的好处，即：“**清晰明了**”。这样的做法可能在 `6` 个参数以内都还能**便于调用者一眼知道方法是干什么、需要些什么数据**。

但是在一些特殊复杂的业务场景中，我们其实经常会遇到超过 `6` 个甚至更多的参数的场景，那么：**如果需要的参数超过了 6 个甚至更多呢？？如果后续该业务又要继续增加参数，怎么搞？？**

这种情况就只能再重载一个方法，实现增加 1  个参数的情况。但是...如果后续参数列表逐渐增多，那不就意味着会有一堆重载方法，后期维护成本也增加。

其次，**要如何模拟这些数据对象场景，进行编写单元测试代码呢？？？**

因此：**对于复杂业务用通常会用 `Context` 来组合这些 “上下文数据”，并且一般也会提供一个  `buildContext` 的方法来进行统一构建上下文**。

这样的做法可以：

1. **将“业务数据”和“方法实现”进行解耦，便于模拟数据（易于编写单元测试）**
2. **简化形参列表，方便数据传递，实现方法间数据共享**


### 问题3、什么时候用“Context”，什么时候用形参列表传递参数？

1. 传递的业务数据并不复杂
2. 已经很明确：形参列表就这么些，业务变动也很小（不会出现频繁加减参数）
3. 不会出现同一份形参在 A 方法需要，然后加工处理之后又需要传递给 B，然后加工处理又往下传递...这类情况

其实符合上述情况，那么通过 `(obj1, obj2, obj3)` 感觉也是没关系...但如果你预测业务所需要的数据会逐渐的复杂，那么就应该提前考虑是否需要使用 `context` 来传递

### 问题4、“Context” 增加属性时的注意点

当需要增加属性到“上下文”数据对象中时，应当要考虑清楚：增加的这些独立属性，是否是独立于当前任务的业务的？

如果可以独立于业务之外，那么就不应该将该属性直接加入到“上下文”数据对象中，而是将处理加工这些属性后得到的结果作为一个独立的属性增加到“上下文”中。

举例：在 “业务经历” 段落中写到的，我需要提供一个获取车辆保险服务的接口，其中会要求调用方传递「车辆套餐信息（共64个字段，需要用到其中 34 个字段）」、「用户信息」、「上层调用标识」... 等信息

其中「用户信息」、「上层调用标识」这两个属性对象，在保险业务中其实用处极小：

- 用户信息：只是为了获取到用户保险的复购次数，用于最终计算保险的折扣价格
- 调用标识：只是为了通过调用标识用于判断，保险的售卖是否需要判断车辆套餐的原始收款方式

基于此，请思考：

1. 将 「用户信息」整个对象作为「Context」的依赖属性和直接将 「复购次数」这个值作为 1 个属性作为「Context」的依赖属性好呢？哪一种更加方便模拟数据？而且将「根据用户信息获取保险复购次数」这个作为一个独立逻辑抽离出来是否会更加通用且更加方便加缓存呢？
2. 将 「调用标识」作为一个值加到「Context」的依赖属性中和直接将 「是否判断原始收款方式」这个结果值作为属性作为「Context」的依赖属性哪个更好呢？哪一种更加方便模拟数据？如果后续有其他“调用场景”也需要进行判断，那么哪一种处理方式更加符合开闭原则？


## 附录：什么是“上下文”？

大家读书时肯定都做过“阅读理解”这种题目，对于 “请联系上下文，分析xxx的意思” 这种题目肯定都记忆深刻。这种题目中的 “上下文” 其实指的就是 “语境、语意” 的意思，即：根据 xxx 所在段落的前后语句或前后的段落，分析 xxx 在当前语境中到底是要表达一个什么意思？

例如：请联系上下文，分析“啊”是什么意思？

- 例子1：李白费尽了艰辛登上了庐山，发出感叹：“啊，真壮观呢，俺准备作诗一首”
- 例子2：李白费尽了艰辛登上了庐山，惋惜道：“啊，TMD...就这？？？再也不来了”

「例子1」 中的 “啊”，上文是 “感叹” 下文是 “真壮观，准备作诗一首”，很明显这个 “啊” 是正向的反馈，表示至少李白是开心的。

而「例子2」 中的 “啊”，上文是 “惋惜” 下文是 “TMD...就这？？？”、“再也不来了”，很明显这个 “啊” 是负反馈，表示李白对自己 “费尽艰辛” 之后看到的东西有些失望甚至是愤怒...


从语言学科中引申过来到计算机学科中来就是指：**执行一个任务所需要的相关数据及环境信息。**

具体的一个例子：有这么一个伪代码程序（任务），根据应聘者信息（对象）输出录用情况

```java
录用判断 (上下文 对象) {
    if 对象 == null
        抛出异常：什么玩意儿？？？
    else if (对象.成绩A > 对象.成绩A考核标准 且 对象.成绩B > 对象.成绩B考核标准) 
        输出：德才兼备，重要
    else if (对象.成绩A > 对象.成绩A考核标准 且 对象.成绩B < 对象.成绩B考核标准)
        输出：有才无德，慎用
    else if (对象.成绩A < 对象.成绩A考核标准 且 对象.成绩B > 对象.成绩B考核标准)
        输出：有德无才，可用
    else
        输出：无德无才，人道毁灭吧...    
}
```

这个 “对象” 就是一个“上下文”，其包含了“录用判断”这个任务执行时所需要的一切信息。
