499的排查思路总结（麻烦老师帮忙看看是否正确）：

1、前/后端设置超时不对导致。

譬如后端某接口后端需要3秒，而前端设置的请求超时时间小于3秒，那么会导致大量请求被客户端主动关闭，从而引发大量 NGINX 499。一般根据业务重设前端的请求超时时间即可，这种情况下：单独窗口访问接口、LB机器、后端机器、数据库机器等...负载、慢日志可能都是显示都正常的。
该情况一般多见于新对接功能对整体超时时间不熟悉的情况下，感觉倒是不太常见。

2、LB 机器问题导致
在 LB 机器上手动的请求一下要访问的接口，LB代理异常或网络抖动也会导致LB通过内网请求后端服务时超时，从而引发客户端主动关闭请求，导致 LB Nginx 大量的 499。
这个场景，多数由于网络抖动或运维人员进行网络调整之后导致。

3、后端服务异常导致

这个是情况最复杂多样也是最难排查定位的，因为这种情况并单纯是 499 往往还伴有随之而来来的 502/503/504 情况。

一般来说：前端请求超时设置 > LB请求后端超时设置 > 后端服务超时设置，但是大部分应该都不会令 “前端请求超时设置 > LB请求后端超时设置”，因为前端请求超时设置是根据用户不同业务场景设置不同超时时间的，而 LB请求后端的超时时间 是针对所有转发请求的。

若后端服务中的某一环出现硬盘/CPU/内存等资源不足，在高流量情况下会产生流量堆积，从而呈现出：后端先出现大量的 504，LB 机器间歇的出现大量504和499，最后出现大量的 502/503 的情况。（这种情况是否应该：先重启一下服务重置一下资源的占用情况，保证后续的服务能正常响应，缩短影响时间？？）

1). 人为因素：查看网络出入口带宽情况，看看在最近的时间内站点是否有大量的流量涌入导致了流量堆积，从而导致机器负载飙升。
①. 恶意攻击：有日志平台可以快速看一下是否是同一个IP或UA，针对IP/UA迅速屏蔽掉对应的请求。
②. 正常流量：新接入了流量渠道导致流量一下子增加。有没有限流措施可以实施？能否快速横向扩增机器来抗住流量的涌入？如果都不行，先暂时下掉流量渠道

2). 非人为因素：自底向上排查到底是服务哪一环出现问题？
数据库-->缓存-->业务机器，因为下层问题都会导致上层受影响，如果一开始就从上层排查，情况会特别棘手。
①. 数据库机器负载：查看是否有频繁的IO写入操作？查看是否索引问题导致大量慢日志？是否因为缓存雪崩/缓存穿透/缓存击穿，导致缓存失效从而流量查询直接到了数据库？
②. 缓存机器负载：IO/内存情况-看key的使用效率和IO操作？big key问题导致内存不足？往往是缓存的数据结构设计不正确导致
③. 业务机器负载：是否有代码逻辑不合理，导致业务机器有大量的IO处理、内存溢出...等情况？

非人为因素，没有固定的解决办法，往往需要在各个环节都做出调整，譬如：数据库添加合适的索引、针对热点查询增加正确的缓存结构、业务代码进行调整（如：避免在循环中查询数据/缓存）

- OSI 七层模型是理论参考模型，简单来说就是：华而不实-理论很好，但应用性不足。TCP/IP 五层模型只是为了介绍网络原理而设计的，实际应用还是 TCP/IP 四层体系架构。
- 一般来说，TCP/IP体系架构中的应用层的协议，我们主要还是面对OSI七层模型中的应用层、会话层两层中比较多。

## 02 | 抓包分析技术初探：你会用tcpdump和Wireshark吗？

- tcpdump 依赖底层能力：BPF（全称：Berkeley Packet Filter，又称：BSD Packet Filter. 它是一个内核态追踪工具）
- BPF 全称是 Berkeley Packet Filter（也叫 BSD Packet Filter），它以一种新的基于寄存器的虚拟机方式，实现了用户程序高效稳定的和系统内核交互，从而实现报文过滤。
- libpcap 是 BPF 的一层 API 封装，使的用户空间程序（tcpdump、Wireshark等）可以方便调用 BPF 实现抓包和过滤功能。
- WinPcap 是 Windows 版本的 libpcap
- eBPF（extended BPF）更加通用的内核接口，不仅能网络抓包过滤，还能支持网络以外的内核观测点的信息收集工作；更因其在数据面上的性能很出色，所以也可用它实现一些数据面的开发工作，如高性能的负载均衡。
- libpcap 抓包文件格式 `*.pcap`；`*.pcapng` 是弥补 `*.pcap` 格式不足所产生的新格式，单个抓包文件可包含多网口报文信息；`*.cap` 是一些其他抓包程序生成的抓包文件，用 Wireshark 正确的版本是可以读取这些信息的。

## 03 | 握手：TCP连接都是用TCP协议沟通的吗？

- 客户端发起连接强强可能因为各种原因导致没有回复，此时客户端发起重试。在 Linux 系统中，默认情况下 TCP 客户端会发起 6 次重试，该重试次数可通过内核参数 `net.ipv4.tcp_syn_retries` 设置（[Linux 内核参数修改](https://www.cnblogs.com/diantong/p/11287836.html)）。重试之间遵循**指数退避**原则。即：第 N 次重试需要等待距离上一次请求的时间公式如下

```math
WaitToRetryTime = 2^n 【其中：n=N-1】
```

- TCP 服务端拒绝握手会通过 ICMP 报文回复拒绝。通过查看 Iptables 拒绝的规则设置，发现被自动添加了 `--reject-with icmp-port-unrechable` 选项（`--reject-with tcp-reset`）。
- `nc` 命令测试连接性：对于 UDP 来说，除非收到明确的拒绝报文，否则就视为“连通”；对于 TCP 来说，除非收到明确的接受，否则就视为“不可连通”。
- [一台机器最多 65535 个 TCP 连接吗？](https://mp.weixin.qq.com/s/X6c_H5_4OInR8nFQVn7IMA)错误的，单机情况仅在设置了一个出口IP和一个服务端口时成立（同一个源IP的同一个端口可接受连接范围 `/proc/sys/net/ipv4/ip_local_port_range`）。[知乎：一台主机上只能保持最多 65535 个 TCP 连接吗？](https://www.zhihu.com/question/361111920)
- 客户端和服务端能同时握手吗？可以！见 [TCP 三次握手、四次挥手](https://segmentfault.com/a/1190000039165592)
- **FAQ：为什么指数退避原则本身就不建议在整秒做重试？**
    - 最好是随机时间，或翻倍时间附件有一定的浮动，这样避免每次都发生资源占用撞车。对于一般的应用逻辑也是一样（好比：我们设置秒级定时任务的时候，秒数一般都随机选择一个，而不要总集中在 10/15/30 这类常用秒数）

## 04-挥手：Nginx日志报connection reset by peer是怎么回事？

- `upstream` 上游：在网络运维的视角，更关注网络报文的流向（HTTP 报文从 `Client(upstream)-->Proxy-->Server`）；在应用的视角上，我们更关注数据的流向（数据是从：`Server(upstream)-->Proxy-->Client`）。Nginx、Enovy属于应用网关，因此配置时 `upstream` 是指服务端。
- 学会如何使用 Wireshark 的使用和过滤器规则的书写：《Wireshark 抓包就是这么简单》
