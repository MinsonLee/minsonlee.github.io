### **4、关于 `Dockerfile workdir` 关键字的疑问，发现对这个有一点点小模糊，记录一下**

群友-冯建宏在通过 Dockerfile 构建 nginx 镜像时遇到的问题

```yaml
...
ADD nginx-1.21.6.tar.gz /usr/local/src
WORKDIR /usr/local/src
WORKDIR nginx-1.21.6
...
RUN ...
...
```

问，这里两个 `workdir` 后续是用的哪个？

1. 问题忽略了前面一行的 `ADD` 命令，`ADD` 命令和 `COPY` 命令的使用格式和性质基本一致，但是 `ADD` 命令在 `COPY`-复制文件 的基础之上增加了一些功能。
    - `ADD <URL> <tag-path>` - 尝试自动下载 URL 的文件放到目标路径中去，并自动将下载后的文件赋权为 `600`
    - `ADD <*.tar.*> <tag-path>` - 如果是一个 tar 归档文件且文件压缩方式为：gzip、bzip2、xz 的情况下，`ADD` 会拷贝文件并将文件自动解压缩到目标路径中去
2. `WORKDIR <source> <tag>` 的 `<tag>` 可以是一个绝对路径或基于当前工作目录的相对路径，如果目录不存在就会自动创建目录并 cd 进去（当时错误的记为了相对路径都是基于 `~` 目录创建再 `cd` 了）

所以上述群友的问题，最终工作目录是 `/usr/local/src/nginx-1.21.6`

### **5、删除 K8S 中的 namespace 卡住**

 1. 根据 https://www.cnblogs.com/elnino/p/11104084.html 操作，依然遇到了问题

```sh
[root@master0l calicodir]# kubectl api-resources --verbs=list --namespaced -o name | xargs -n 1 kubectl get --show-kind --ignore-not-found -n tigera-operator
error: unable to retrieve the complete list of server APIs: metrics.k8s.io/vlbetal: the server is currently unable to handle the request

[root@master0l calicodir]# kubectl delete ns --force tigera-operator
warning:Immediate deletion does not wait for confirmation that th he running resource has been terminated. The resource may continue to ru n on the cluster indefinitely.
namespace "tigera-operator" force deleted
...
```

2. 张老师建议：最好先去删除下当前 `ns` 中的所有资源，或者：
`kubectl delete namespace tigera-operator --grace-period=0 --force`
`kubectl get namespace tigera-operator -o json | jq '.spec.finalizers = []' | kubectl replace --raw /api/v1/namespaces/tigera-operator/finalize `

使用强制删除命令可能会导致数据丢失，因此请谨慎使用。仅作为测试使用，线上不要冲动。

3. 群友-默念阿门给了亲测建议：有时候 `ns` 下面没有任何资源，就是删不掉，一直`terminating`，且可能`--force`和`json`方式也不好使，可以使用 `kubectl edit ns` 修改一下 `yaml` 文件，把里面的 `finalizers` 字段删掉就可以了，这种方式最彻底，但慎用。

对 `Finalizers` 资源对象的理解：[官方文档](https://kubernetes.io/zh-cn/docs/concepts/overview/working-with-objects/finalizers/)


### **！！！6、k8s 集群启动失败排查思路**

群友-清风 在安装 K8S 集群（一主一从/master01-worker01）时遇到问题，`kubectl get nodes` 查看 `worker01` 节点一直起不来，通过 `kubectl get pods --all-namespace` 查看有个 Pod 一直启动不起来，群里大家讨论给予的建议、思路可借鉴

![启动 k8s 集群失败](https://cdn.jsdelivr.net/gh/MinsonLee/minsonlee.github.io/images/pig/k8s-init-pod-fail.png)

1. `kubectl get pods -A` 查看存在问题的 `Pod`
2. `kubectl describe pod <pod-name> -n <namespace>` 查看指定 pod 资源的详细状态，注重看 `Events` 的事件信息（1、2 是通用的步骤）
3. 通过上述两步发现是 Pod 在拉取镜像 `k8s.gcr.io/pause:3.4.1` 的时候报错了，原因是连接 `k8s.gcr.io` 域名超时了
4. 问题基本锁定，接下来是：确认问题 + 选择解决方案：

因为部署时如果在**本机**没有发现对应 TAG 的镜像，则会去对应的 Container Registry 拉取镜像，所以解决思路有两个：
- ①. 确定镜像是否可以访问，如果可以那么解决网络问题即可
- ②. 去 阿里云/Dockerhub 或其他镜像仓库 拉一个同样版本的镜像下来，然后修改 tag

> 通过 `docker tag IMAGE[:TAG] [REGISTRYHOST/][USERNAME/]NAME[:TAG]` 新建一个镜像 tag

5. 确认问题
- ①. 确定有问题的 Pod 在哪个节点上：`kubectl get po -A -o wide`
- ②. 确认该节点上是否有对应 Tag 的镜像： `docker images | grep pause`

清风被问题卡住，关键就在 5-① 这一步...因此他自己没能解决问题。因为他所有操作都是在 `master` 节点上操作，通过  `docker images | grep pause` 看到 `master` 节点已经执行过 `docker tag registry.cn-hangzhou.aliyuncs.com/google_containers/pause:3.4.1 k8s .gcr.io/pause:3.4.1` 且存在，因此不明白为什么拉取镜像会出现超时、错误。

最终去到 `worker01` 节点重新对镜像打个 Tag 即可，整个集群就启动起来了。

群里反馈的其他问题，排查思路基本目前都是类似套路。

- 群友给的安装配置建议
```txt
企业生产环境：
a) Master节点：三个节点实现高可用（必须）
i.节点数：0-100    8核16+
ii.节点数：100-250  8核32G+
iii.节点数：250-500  16核32G+

b) etcd节点：三个节点实现高可用（必须），有条件存储分区必须高性能SSD硬盘，没有SSD也要有高效独立磁盘
i.节点数：0-50    2核8G+   50G SSD存储
ii.节点数：50-250  4核16G+  150G SSD存储
iii.节点数：250-1000  8核32G+ 250G SSD存储

c) Node节点：无特殊要求，主要是Docker或Containerd的数据分区、系统分区需要单独使用，不可以使用同一个磁盘，系统分区100G+、Docker或Containerd数据分区200G+，有条件使用SSD硬盘，必须独立于系统盘

d) 其他：集群规模不大可以将etcd和master放置于同一个宿主机，也就是每个master节点部署k8s组件和etcd服务，但是etcd的数据目录一定要独立，并且使用SSD， 两者部署在一起需要相对增加宿主机的资源，个人建议生产环境把master节点的资源一次性给够，此处的费用不应该节省，可以直接使用16核32G或者64G的机器，之后集群扩容就无需扩容master节点的资源，减少风险。其中master节点和etcd节点的系统分区100G即可。
```
