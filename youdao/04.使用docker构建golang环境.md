

由于实训营的伙伴都在问曹大对 `Golang` 的版本要求问题，所以曹大直接给了一个基于 `CentOS` 做基础镜像的 `Docker` 镜像出来。但是编译完之后我发现有些太大了，让我本就贫瘠的磁盘空间不得不删了一部电影才装下它。

`DockerHub` 上 `Golang` 的官方镜像目前只有 `1.15.X` 的版本，而且没有预装 `gdb`、`dlv`这些上课必须要必备的东西。所以我自己重写了一个基于 `Alpine` 的上课需要的 `Dockerfile` 文件

1、Dockerfile 文件如下

```Dockerfile
# 曹大提供的 Dockerfile
FROM centos
RUN yum install dlv -y \
    && yum install binutils -y \
    && yum install vim -y \
    && yum install gdb -y \
    && yum install git -y \
    && yum install wget -y

RUN wget https://dl.google.com/go/go1.14.12.linux-amd64.tar.gz  \
    && tar -C /usr/local -zxvf go1.14.12.linux-amd64.tar.gz \
    && rm /go1.14.12.linux-amd64.tar.gz

ENV PATH /usr/local/go/bin:$PATH
ENV GOPATH /home
ENV GOROOT /usr/local/go
ENV GOPROXY goproxy.cn
```

基于 `Alpine` 且安装了 `gdb`、`dlv` 的 `Docker` 镜像构建文件

```Dockerfile
# Dockerfile 文件

# 基于 alpine 最新版本作基础镜像
FROM alpine:latest

# 作者名称
MAINTAINER limingshuang lms.minsonlee@gmail.com

# 环境变量
env GOPATH /home
env GOROOT /usr/local/go
env GOPROXY goproxy.cn
env PATH $GOROOT/bin:$GOPATH/bin:$PATH

# 构建镜像
# 1. 修改 alpine 镜像源为阿里云镜像源
RUN sed -i 's/dl-cdn.alpinelinux.org/mirrors.aliyun.com/g' /etc/apk/repositories \
    # 2. 安装基础软件和依赖
    && apk add --no-cache --virtual .build-deps wget git tar \
    && apk add --no-cache gdb binutils libc6-compat \
    # 3. 下载 golang 1.14.12 && 解压
    && wget -O /tmp/go.tar.gz https://dl.google.com/go/go1.14.12.linux-amd64.tar.gz \
    && tar -C /usr/local -zxvf /tmp/go.tar.gz \
    # 4. 安装 dlv
    && echo "140.82.112.4 github.com" >> /etc/hosts \
    && echo "185.199.108.154 github.githubassets.com" >> /etc/hosts \
    && echo "199.232.69.194 github.global.ssl.fastly.net" >> /etc/hosts \
    && git clone https://github.com/go-delve/delve /tmp/delve \
    && cd /tmp/delve/cmd/dlv \
    && go build && go install \
    # 4. 清理无用安装文件和缓存文件
    && rm -rf /tmp/* \
    && rm -rf $GOPATH/pkg/* \
    && rm -rf /var/cache/apk/* \
    && rm -rf /var/src/* \
    && apk del .build-deps

# 设置工作目录
WORKDIR $GOPATH

STOPSIGNAL SIGQUIT

```

源码见： [GitHub](git@github.com:MinsonLee/alpine-docker.git)

![HTTP/2 stream 1 was not closed cleanly before end of the underlying stream](https://cdn.jsdelivr.net/gh/MinsonLee/minsonlee.github.io/images/pig/git-error-http2-stream-1-was-not-closed.png)

在构建安装 dlv 的过程中，报错：`fatal: unable to access 'https://github.com/go-delve/delve/': HTTP/2 stream 1 was not closed cleanly before end of the underlying stream`。

通过网上查了一下发现是 Git 调用 cURL 时候出现的问题（见：[getting-curl-92-http-2-stream-1-was-not-closed-cleanly-internal-error-err](https://stackoverflow.com/questions/56413290/getting-curl-92-http-2-stream-1-was-not-closed-cleanly-internal-error-err)）。Git  默认使用的 HTTP 通信协议出现了问题，可以通过将默认通信协议修改为 http/1.1 来解决该问题。

```shell
git config --global http.version HTTP/1.1
```

## 问题参考

### 一、报错：`sh: go: not found`

我第一次按照曹大的 Dockerfile 结合 golang 官方提供的 Dockerfile 进行构建镜像，启动容器之后发现 `go` 命令一直提示 `sh: go: not found`。报错如下：

```sh
3b1990414422:/tmp# go version
sh: go: not found

3b1990414422:/tmp# echo $PATH
/usr/local/go/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

3b1990414422:/tmp# source /etc/profile

3b1990414422:/tmp# go version
sh: go: not found

3b1990414422:/tmp# /usr/local/go/bin/go version
sh: /usr/local/go/bin/go: not found
```

**解决方案**

不管我怎么刷新环境变量 `PATH` 且再三确认了环境变量的 `PATH` 路径一定是没有问题的，可在全局调用 `go` 命令或通过绝对路径 `/usr/local/go/bin/go` 调用 时依然报错，所以怀疑应该是依赖包安装遗漏了。

1、使用 `ldd` 查看调用 `go` 时所依赖的扩展包

```sh
3b1990414422:/tmp# ldd /usr/local/go/bin/go
        /lib64/ld-linux-x86-64.so.2 (0x7ff16eb34000)
        libpthread.so.0 => /lib64/ld-linux-x86-64.so.2 (0x7ff16eb34000)
        libc.so.6 => /lib64/ld-linux-x86-64.so.2 (0x7ff16eb34000)
```

发现调用时依赖 `/lib64/ld-linux-x86-64.so.2` 这个扩展，一开始以为是这个扩展没有权限，结果一查看：这个扩展居然没有被安装。

```
3b1990414422:/tmp# ls -al /lib64/ld-linux-x86-64.so.2
ls: /lib64/ld-linux-x86-64.so.2: No such file or directory
```

去 `Alpine` 官网搜索该扩展所依赖的包，然后安装对应的 `libc6-compat`，发现问题就得到解决了。

### 二、容器 `hosts` 文件初始化问题

由于在国内访问 `github.com` 会很慢，因此在使用`Dockerfile`构建基于`Alpine`的`Golang + dlv`镜像时，我特意将加速访问的 `IP/域名` 添加到 `/etc/hosts` 中，但当我构建完成之后启动容器时发现 `/etc/hosts` 被重新初始化覆盖了。

查找了一下书籍关于容器网络的篇章，是因为：`/etc/hosts` 这个文件实际在容器启动的时候才会被生成，因此在构建过程中添加的信息是无用的。

如果我们想添加`hosts`，可以通过 `entrypoint` 方式，在容器启动时执行添加；或者，通过 `--add-host=domain:ip` 的方式添加。

```sh
docker run -it -v /data/golearn:/home \
--add-host=github.com:140.82.112.4 \
--add-host=github.githubassets.com:185.199.108.154 \
--add-host=github.global.ssl.fastly.net:199.232.69.194 \
--name golearn-host golearn
```

此时在容器内通过 `cat /etc/hosts` 可以看到对应 IP/域名 被成功添加到文件中

### 三、`Alpine` 通过 `apk add` 安装扩展包 `--virtual NAME` 的意义

`Alpine` 使用 `apk` 命令进行依赖扩展包管理，在基于 `Alpine` 作为基础镜像构建 `Alpine+Golang + dlv` 镜像的 `Dockerfile` 中，我最开始构建出来的镜像是 521M。

![image](D04E85F297CC403392977ACD8A93C8AD)

虽然比其最开始的镜像已经小了 300M+，`golang 1.14.12` 的源码包解压完成之后大约是 380M+，也就是说其余的扩展包也装了近 140M。

之前曾看过部门本地环境的 Dockerfile， 我记得曾看到过类似“清理安装现场”的操作，对比了一下发现了使用 `apk` 进行安装时有一个 `--virtual NAME` 的操作。

官方手册的解释如下：

```sh
$ apk add --help

-t, --virtual NAME    Instead of adding all the packages to 'world', create a new virtual package with the listed dependencies and add that to 'world'; the actions of the command are easily reverted by deleting the virtual package
```

简答来说就是将多个依赖包合并成一个重命名的新的依赖包，管理操作时只需要操作这个新的本地依赖包即可，方便一次性恢复或删除这多个包。

例如：将 `gcc`和`vim`两个扩展涉及的所有依赖包打包为一个扩展，并将其命名为 `mypacks`，这样在使用完之后可以一次性卸载或进行恢复安装

```sh
apk add --virtual mypacks gcc vim
apk del mypacks # 相当于执行 apk del  gcc vim
```

我们在使用 `alpine` 作为基础时就可以利用这一特性，针对使用 `apk` 安装扩展，在镜像构建即将完后前卸载所有运行代码过程中不需要使用到的依赖，从而达到减小镜像体积的作用。
