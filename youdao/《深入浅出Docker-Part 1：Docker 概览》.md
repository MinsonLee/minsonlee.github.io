## 第1章：容器发展之路
### 容器为什么出现？
- **旧时代**：采用单机单应用部署方式，而购买的机器往往都是比应用实际所需更大、更好的服务器，该做法导致了==大部分的服务器长期运行在他们额定负载 5% - 10% 的水平区间内==，造成了公司资产和资源的极大浪费。


- **VMware**：虚拟机(VM)的出现具有划时代的意义，==使的"单机多应用"能够稳定、安全的同时部署运行==，让现有的资产(特别是服务器资产)拥有了更大的价值，节省了大量的资金支出
> 虚拟机的不足：依赖其专用的操作系统(OS)，由 OS 维护带来的运营成本(OPEX)和资金性支出(CAPEX)带来的问题日益凸显。
> - “虚拟机”是通过软件模拟的具有完整硬件系统功能的、运行在一个完全隔离环境中的完整计算机系统。因此部署在虚拟机上的应用对操作系统(OS)的依赖非常大；
> - OS 会占用宿主机额外的 CPU、RAM及存储，而这些资源本可以运行更多的应用
> - 每个虚拟机都有单独的 OS，意味着维护成本更大，如：每个 OS 都需要补丁和监控；某些情况下 OS 是需要购买许可证的，独立的 OS 意味着需要购买多个独立的许可证


- **使用容器(Container)**：容器模型跟虚拟机模型相似，主要的区别在于==容器的运行不会独占操作系统==。由于容器技术的复杂性，长期以来，只有像谷歌、亚马逊、围绕这样的大规模Web服务玩家才一直采用容器技术解决虚拟机模型的缺点。
> - ==多容器共享同一个操作系统==：运行在相同宿主机上的容器是共享一个操作系统的，这样就能节省大量的系统资源，如：CPU、RAM 以及存储，更好的解决了虚拟机带来的问题（OS 的维护、许可证支出...）
> - ==启动迅速==：容器具有启动速度快！由于不需要额外虚拟分配系统资源，所以容器的启动速度得到了大大的提升
> - ==迁移交付完整==：容器将操作系统、程序库、配置文件、路径和应用程序打包在一起运行，也就是说，我们交付测试时整个镜像是什么样，迁移到产品环境以后就是什么样，其性能不会有任何差距。


- **Linux 容器（Linux Container-简称：LXC）**：现代容器技术起源于Linux，08年出现诞生。Linux Container主要由Namespace和Cgroup两大机制来保证实现，使的容器之间的独立和分层；联合文件系统(Union File System)使的更大成本的节省了存储资源；
> 但是此时的容器技术成本仍然是巨大的，技术人员需要自己去关注、搭建、使用这些依赖的底层技术去使用容器

- **Hello，Docker**：Docker 技术的出现，使的容器技术变得更加的简单。它对 LXC 进行了封装，使的开放人员可以忽略复杂的底层技术，更加专注的关心使用上即可！
> - Docker 技术的出现，极大的推动了容器的普及，但随之而来的问题也来临了：如何管理监控(编排)诸多容器？


- **Kubernetes**：Kubernetes 是谷歌用于容器编排的一个开源项目(Kubernetes 是保证容器部署和运行的软件体系中很重要的一部分)
> - Getting Started With Kubernetes：https://www.youtube.com/watch?v=GyRqtyAz5l8v
> - Kubernetes Doc：https://kubernetes.io/docs/tutorials/hello-minikube/

## 第2章：走进 Docker
- Docker 公司：位于旧金山，原名 dotCould，是一家容器平台服务提供商，为了方便创建和管理这些容器，其开发了一套内部工具(即：Docker)。是 Docker 项目的主要官方维护者，公司 13 年更名为 Docker。


- Docker 项目：在 Github 上开源的 Docker 项目，17年项目更名为 Moby(项目的目标是：基于开源的方式，发展成为 Docker 上游，将 Docker 拆分为更多模块化组件)。


- Docker 引擎（Docker）：大多数技术人员谈论 Docker 的时候，主要都是指 Docker 引擎。Docker 引擎是用于运行和编排容器的基础设施工具。
