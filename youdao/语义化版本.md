# 语义化版本控制

[TOC]

混迹互联网的人应该都知道**版本号**这个概念，譬如我们经常说的：微信可以更新到 `8.0.7` 了。这里的 `8.0.7` 就是指微信的版本号.

如果你刚好是程序员，那么版本号更加和你密切相关。或多或少你也应该听过**语义化版本格式**这个名词。语义化版本格式如下：**主版本号.次版本号.修订版本号**

例如：你提交代码打 Tag 的时候、在 GitHub 上克隆代码的时候、安装软件的时候、代码中添加第三方库依赖的时候...等等，这些都能随处可见版本号的概念。

特别是在代码中添加第三方库依赖的时候，如果添加的依赖库版本随意命名版本，那么“**什么时候应该去更新依赖的版本？**”这个问题，简直就是噩梦。

一般来说：开源代码中，我们常见版本号是：**`X.Y.Z`** 这样的格式；开源软件中，我们常见的版本号格式是：**`XXX X.Y.Z.date_beta`** 的样式。

## 语义化版本格式  

- **X**-主版本号：当你做了不兼容的 API 修改（即：两个版本之间差异大，且不兼容。一般引用外部包或第三方库，主版本号都会固定不会随便乱升级）
- **Y**-次版本号：当你做了向下兼容的功能性新增（即：API 有新增或修改，但是**向下兼容**-不影响低版本运行）
- **Z**-修订号：当你做了向下兼容的问题修正（即：仅问题修复，必须***向下兼容**，无功能的新增或修改，低版本的调用不需要调整）
- **date**-日期版本（先行版本号）：一般是用来记录项目修改日期，如果说你每天都要更新版本那么可能会用到该版本号位

绝大部分的版本号都是以上 **`X.Y.Z`** 的格式，**`date`** 版本号应该是比较少见或我不常见到的。

- **X、Y 和 Z 为非负的整数，且禁止（MUST NOT）在数字前方补零**
- **标记版本号的软件发行后，禁止（MUST NOT）改变该版本软件的内容。任何修改都必须（MUST）以新版本发行**
- **主版本号为零（0.y.z）的软件处于开发初始阶段，一切都可能随时被改变。这样的公共 API 不应该被视为稳定版**
- **1.0.0 的版本号用于界定公共 API 的形成。这一版本之后所有的版本号更新都基于公共 API 及其修改内容**
- **每当主版本号-X 递增时，次版本号和修订号必须（MUST）归零**

正确示范 | 错误示范
---|---
1.10.0 | 1.10.-1
1.10.1 | 1.10.01
1.10.1 --> 2.0.0 | 1.10.1 --> 2.0.1

### 发行版本/先行版本格式

还有些安装软件的版本，末尾可能还有一些字母版本，如：`1.0.0-beta`、`1.0.0-alpha`、`1.0.0-x.7.z.92` 之类的，这类版本标识为先行版本。

- **`alpha`**：内部测试版本，不对外开放，Bug 多
- **`beta`**：测试版，没啥致命问题，可能有些小缺陷
- **`lts`**：长期维护版本，放心用，在没通知不维护之前都有后勤部队服务
- **`rc`**：即将作为正式版发布，已经基本成熟，可能还会有些小优化
- **`release`**：发行版，最终交付的产品


**先行版本号可以（MAY）被标注在修订版之后，先加上一个连接号`-`再加上一连串以点号分隔 `.` 的标识符`[0-9A-Za-z]`来修饰**。

当进行版本比较时，**==当主版本号、次版本号及修订号都相同时，改以优先层级比较低的先行版本号决定。例如：`1.0.0-alpha` < `1.0.0`，`1.0.0-alpha.1` < `1.0.0-alpha`==**

## 唠叨几句

一般来说，开发者使用依赖包：`X` 都会固定不变；`Y` 一般让其保留更新，但因为该版本会增加功能所以也有人选择不更新；`Z` 和 `date` 基本都是保持最新的版本。

我曾经在一个技术群里碰到有人问过：**主版本号加到了 99 不够用了怎么办？**，其实命名版本号是一个学问。

1、如果真的迭代速度太快，那么应当考虑引入 `date` 版本号进行辅助；

2、开发要有责任感和前瞻性的问题，这需要我们在写代码的时候深思熟虑，尽量让我们自己的代码和软件版本能够向下兼容，让使用者能最小代价的升级或不升级；

3、如果真的每次都需要提升大版本号来处理，那么可以考虑换人

关于常规语义化版本控制的规范，是有明确的文档进行规定的。
