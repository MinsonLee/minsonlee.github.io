## Golang 概述
### Go 语言的 SDK 是什么？
- `SDK` 是 `Software Development Kit` 的缩写，即：软件开发**工具包**。Go 开发，首先需要先安装并配置好 SDK
- `API` 是 `Application Programming Interfaces` 的缩写，即：应用程序接口，API 就是可以轻松实现和其他软件的交互

区别：
- API 是一个有**特定的功能**的**数据接口**，功能简单、方便快捷
- SDK 是一个**很多功能构成的集合体**，相当于开发集成工具环境，功能负责、功能齐全

> [API 和 SDK 的区别是什么？](https://mp.weixin.qq.com/s/EkBeT5g9IcSeMvffI-NVxQ)

### Golang 环境变量配置及其作用？
- `GOROOT` 指定 go sdk 安装目录【实际不设置似乎也没什么影响】
- 环境变量 `PATH` 加入 `$GOROOT/bin`(Windows下是 `%GOROOT%/bin`)目录，便于全局使用：`go` `gofmt` `godoc`的命令
- `GOPATH` 指定 Golang 的工作目录(项目目录)，便于 Golang 寻找安装包

### Golang 程序的编写、编译、运行步骤是什么？能否一步执行？
- 编写：开发写源码
- 编译：通过`go build`命令可以将 Golang 的源码编译生成为可机器识别的**二进制可执行文件**
- 运行：Windows 下直接 xxx.exe 文件运行；类 *unix 系统下通过 `./xxx` 进行运行

测试环境可直接通过`go run xxx`的方式，隐式编译源码不直接生成可执行文件直接运行源码

**`go build` 和 `go run` 区别如下：**
- `go build` 会将源码进行编译生成可执行二进制文件，而`go run`则不会生成生成二进制文件
- `go build` 预编译源码时会将程序运行所依赖的库文件包含在生成的可执行二进制文件中，因此生成的可执行文件体积会变大很多
- `go build`生成的可执行文件对 go 开发环境没有依赖，即：将`hello.exe`文件拷贝到没有 go 开发环境的机器上，仍然可以运行
- `go run` 可以直接运行源码，而`go build`则需要多一步手动执行操作
- `go run` 明显是要比直接执行二进制文件要慢的多，因为多了编译器隐式编译自动执行的步骤
- `go run` 直接执行源码的方式对 go 开发环境存在依赖，否则无法执行

为了方便调试，在开发的时候我们可以直接使用 `go run` 的方式进行调试，但在服务器上我们仍然需要使用 `go build` 方式 先将源码进行静态编译为可执行二进制文件，直接运行！

### Golang 程序的编写规则？
- go 文件的后缀必须是 `.go`
- go 是静态编译强类型语言，严格区分大小写
- go 语句后，**不需要**带分号
- go 定义的变量，或 import 包，必须使用，若没有使用就会报错
- go 中，**不要**将多条语句放在同一行，否则会报错
- go 中的花括弧必须**成对出现**，且 `{` 不能独立成行


## 基础数据类型
### 数据类型的转换细节

**1. 以下编译是否能通过？如何修正？**
```golang
func main() {
   var n1 int32 = 12
   var n2 int64
   var n3 int8
   
   n2 = n1 + 20
   n3 = n1 + 20
}
```
1. 编译不能通过
- 原因：变量的类型不因运算操作而改变。即：变量n1 是 int32 类型，加上20之后，仍然是 int32，类型是不会发生改变的
2. 修正如下：
```golang
n2 = int64(n1) + 20
n3 = int8(n1) + 20
```

**2. 下述编译是否会通过？如何修正？**
```golang
var n1 int32 = 12
var n2 int8
var n3 int8

n2 = int8(n1) + 127
n3 = int8(n1) + 128
```
1. 编译不通过
- 变量 n2 编译不会报错，但是结果不是 127+12，按溢出处理
- n3 编译不通过报错`constant 128 overflows int8
`，运算操作的两个操作数必须是同一类型，由于 128 已经超出了 int8 的表数范围，且不能通过 int8(128) 进行强制转换，因此编译操作报错
